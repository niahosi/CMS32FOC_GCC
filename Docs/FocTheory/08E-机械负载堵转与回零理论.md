# 机械负载堵转与回零理论

本文从外部理论笔记迁入，并已按当前 GCC 工程路径和参数做同步；若后续代码与本文冲突，以当前源码为准。

本文讲速度模式、机械负载、堵转判断和当前 ScrewAxis 双端回零/行程标定逻辑。

FOC 电流环只负责让 `iq` 跟随；机械轴是否动，还取决于：

```text
电机转矩
负载转矩
摩擦
惯量
机械限位
螺杆传动
```

当前回零代码在：

```text
Firmware/App/screw_axis.cpp
Firmware/App/screw_axis.h
```

## 1. 机械模型

机械系统常见模型：

```text
J * d(w)/dt = Te - Tload - B*w - friction
```

其中：

```text
J = 转动惯量
w = 机械角速度
Te = 电机电磁转矩
Tload = 外部负载转矩
B*w = 粘性阻尼
friction = 摩擦
```

表贴 PMSM 初学可用：

```text
Te ~= Kt * iq
```

所以：

```text
iq_limit 决定最大可用转矩。
速度环输出 speed_iq_cmd，本质是在请求更多或更少转矩。
```

## 2. 堵转时数据长什么样

堵转不是“电机没命令”，而是：

```text
速度接近 0
速度环仍在请求较大 iq
```

当前 ScrewAxis 使用的判断：

```text
abs(speed_fb_rpm) <= kStallSpeedRpm
abs(speed_iq_cmd) >= iq_limit - kStallIqMargin
持续 hold_ms
```

当前量级：

```text
kStallSpeedRpm = 8
kStallIqMargin = 2
kFastStallHoldMs = 100
kSlowStallHoldMs = 250
```

这说明回零判断不是看单个瞬间，而是要求堵转特征持续一段时间。

## 3. 为什么先伸出最大位置，再收回最里面

当前状态机：

```text
SeekOuter
  -> FastRetract
  -> FastBackoff
  -> SlowRetract
  -> FinalBackoff
  -> Done
```

理论目的：

```text
SeekOuter:
  正向伸出到最大外端，先确认整段机械行程的另一端。
  到端点时记录 outer_encoder_pos。

FastRetract:
  从外端快速负向收回，快速找到内端附近，提高效率。

FastBackoff:
  正向离开内端，释放第一次撞击产生的机械压紧。

SlowRetract:
  低速再次负向接近内端，提高零位一致性。
  确认堵转后记录 zero_encoder_pos。

FinalBackoff:
  记录零位后正向离开内端，避免长期顶住机械限位。
```

这种流程不是单纯“收到最里面”，而是先找到外端，再以内端作为零位。
好处是同时得到：

```text
outer_encoder_pos:
  最大伸出端的编码器累计位置。

zero_encoder_pos:
  最里面内端的编码器累计位置，也就是位置零位。

travel_counts:
  outer_encoder_pos - zero_encoder_pos，表示整段有效行程 count。

travel_mm_x100:
  travel_counts 按丝杠导程换算出的物理行程，单位 0.01 mm。
  当前导程 0.50 mm/rev 写成 50，10.00 mm 写成 1000。
```

内端仍然使用“快速接近、退离、慢速精找”的两次接近方法，比“一次高速撞端点就记零位”更稳。

## 4. 为什么用速度模式做回零

当前回零命令写入：

```text
control_mode = Speed
speed_ref_rpm = fast/slow/backoff speed
iq_limit = s_home.iq_limit
```

速度模式的好处：

```text
未碰端点时，电机按目标速度运动。
碰端点后，速度掉到接近 0。
速度环增加 iq 请求，但被 iq_limit 限住。
可以用 speed + iq 判断端点。
```

如果用纯电流模式，系统只能给固定转矩，不容易区分“正在移动”和“已经撞端点”。

## 5. 误判堵转的来源

堵转判断可能误判：

```text
iq_limit 太小:
  负载稍大就速度上不去，被误判端点。

  回零可以故意用较小 iq_limit，因为目标就是轻推找端。
  位置/往返测试不能继续沿用太小的回零电流，否则正常加速也可能被判成堵转。

速度估算低速抖动:
  speed_fb_rpm 在零附近跳。

机械卡滞:
  中途卡住也像端点。

速度方向或编码器方向错:
  speed_fb_rpm 判断失真。

电流环跟不上:
  speed_iq_cmd 到限幅，但实际 iq 没有达到。
```

所以回零时除了看 `g_screw_axis_watch.home_state`，还要看：

```text
g_motor.speed.feedback
g_motor.speed.iq_ref.value
g_motor.current.dq.q
g_motor.output.voltage_limited
g_motor.runtime.fault
g_motor.encoder.pos
```

## 6. 回零参数意义

当前默认：

```text
fast_speed_rpm = -2000
slow_speed_rpm = -50
extend_speed_rpm = 2000
backoff_speed_rpm = 300
iq_limit = 24
timeout_ms = 12000
```

含义：

```text
fast_speed_rpm:
  快速收回找内端速度，内部强制为负向。当前默认 -2000 rpm，用来让前两次粗来回尽快跑完。

slow_speed_rpm:
  精找内端速度，内部强制为负向，越小越稳但越慢。当前默认 -50 rpm，
  明显慢于快速收回。

extend_speed_rpm:
  正向外伸找外端速度。当前默认 2000 rpm，只用于粗找外端，
  不再和精找前退离共用。

backoff_speed_rpm:
  正向退离内端速度。当前默认 300 rpm，用在 FastBackoff 和 FinalBackoff。
  这个速度要明显低于外伸粗找速度，否则精找前退离太猛，后面的 -50 rpm
  低速精找可能需要更久才重新压到内端。

iq_limit:
  回零最大推力。当前用 24，是因为电流环正负 24 已验证能带动丝杠正反运动。

timeout_ms:
  防止外端寻找、快速内收、低速精找一直跑。
```

堵转判据：

```text
abs(speed_fb_rpm) <= 8 rpm
abs(iq_ref) >= iq_limit - 2
快速粗找阶段持续 100 ms
慢速精找阶段持续 250 ms
```

当前程序对速度反馈尖峰做了短暂容忍：

```text
stall_clear_hold_ms = 50 ms
```

含义是：堵转条件已经开始计时以后，如果速度反馈偶尔跳出门槛，不立刻清掉计时；
只有连续约 50 ms 都不满足堵转条件，才认为堵转条件失效。

这样做的原因是 Ozone 5 Hz 看到的是趋势，不是 1 kHz 速度估算的每个样本。内端已经
顶住时，编码器差分速度仍可能因为机械弹性、采样抖动或滤波残差出现尖峰；如果要求
600 ms 完全没有尖峰，低速精找会容易超时。

当前状态值会暴露给 Ozone：

| 状态 | 数值 | 含义 |
| --- | ---: | --- |
| Idle | 0 | 等待 home_enable |
| SeekOuter | 1 | 正向伸出找最大外端 |
| FastRetract | 2 | 快速负向收回找内端 |
| FastBackoff | 3 | 正向退离内端 |
| SlowRetract | 4 | 低速负向精找内端零位 |
| FinalBackoff | 5 | 记录零位后正向退离 |
| Done | 6 | 双端回零完成 |
| Fault | 7 | 回零失败或底层 MotorControl fault |

轴级故障原因：

| fault_reason | 数值 | 含义 |
| --- | ---: | --- |
| None | 0 | 无故障 |
| Motor | 1 | 底层 MotorControl fault |
| HomeSeekOuterTimeout | 2 | 正向找外端超时 |
| HomeFastRetractTimeout | 3 | 从外端快速收回找内端超时 |
| HomeSlowRetractTimeout | 4 | 低速精找内端超时 |

```text
回零阶段堵转:
  是程序主动找机械端点，用它记录 outer_encoder_pos 或 zero_encoder_pos。
```

当前主固件已经删除往返测试入口，ScrewAxis 保留双端回零和 mm_x100
物理位置调试入口。回零用 `g_screw_axis_cmd.home_enable = 1` 触发，完成或
故障后固件会自动清 0；物理位置调试用 `g_screw_axis_cmd.position_enable = 1`
保持使能，后续直接修改 `g_screw_axis_cmd.target_mm_x100`。

回零记录的 `zero_encoder_pos` 是机械内端基准：

```text
SeekOuter 堵转:
  outer_encoder_pos = g_motor.encoder.pos

SlowRetract 堵转:
  zero_encoder_pos = g_motor.encoder.pos

回零完成后:
  travel_counts = outer_encoder_pos - zero_encoder_pos
  pos_counts = g_motor.encoder.pos - zero_encoder_pos
  travel_mm_x100 = travel_counts * 50 / 262144
  pos_mm_x100 = pos_counts * 50 / 262144
  0.50 mm = 50 mm_x100 = 262144 count = 电机一圈
```

当前简化找零版本不再用 `travel_counts` 阻断 `homed = 1`。它只作为观察值，
用来判断两端实际相差多少 encoder count。

注意：这个“绝对位置”是当前上电运行期间的编码器累计 count 快照，保存在 RAM 里的
`s_home.zero_encoder_pos`。断电或复位后不会自动恢复；如果以后要免回零启动，需要再做
非易失参数保存、CRC 校验和启动可信性判断。

当前位置不要直接理解为 MA600 的 0..65535 单圈角度。当前用的是
`g_motor.encoder.pos`，它是 MotorControl 编码器逻辑维护的多圈累计位置；
ScrewAxis 只是把“内端堵转瞬间的累计位置”记下来，后面用当前累计位置减掉这张快照。

## 7. 从数据判断回零问题

### 7.1 一开始就 fault

看：

```text
g_screw_axis_watch.fault_seen
g_screw_axis_watch.fault_reason
g_motor.runtime.state
g_motor.runtime.fault
g_motor.check.current_ok
g_motor.check.ma600_ok
```

先解决底层 MotorControl fault。

### 7.2 中途误判端点

看：

```text
speed_fb_rpm 是否真的接近 0
speed_iq_cmd 是否接近 iq_limit
iq 是否也接近命令
v_limited 是否为 1
encoder_pos 是否还在变化
```

如果 `speed_iq_cmd` 到限幅但 `iq` 没跟上，要查电流环或电压余量。

如果 `iq` 很小但速度为 0，可能是速度环输出没有建立，查速度命令、死区和 enable。

### 7.3 零位重复性差

可能原因：

```text
fast 撞击太大
slow 速度仍太高
iq_limit 太大导致机械压紧
回退距离/时间不足
速度估算低速量化
机械间隙
```

调试方向：

```text
降低 slow_speed_rpm。
降低 iq_limit。
增加 slow stall hold 时间。
增加 backoff 时间。
记录多次 zero_encoder_pos 和 travel_counts 分布。
```

## 9. 学习重点

回零不是单纯电机控制，它是：

```text
速度环
电流限幅
机械负载
堵转判断
状态机
安全保护
```

这些一起工作的结果。调回零前必须先确认：

```text
电流环可信
速度反馈方向正确
速度模式低速可控
故障保护正常
```
