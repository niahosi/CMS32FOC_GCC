# 速度估算与低速控制理论

本文从外部理论笔记迁入，并已按当前 GCC 工程路径和参数做同步；若后续代码与本文冲突，以当前源码为准。

本文讲当前工程的速度估算、低速抖动和速度环调试。

当前速度反馈来自编码器 raw 差分，不是硬件测速外设。

相关代码：

```text
Firmware/MotorControl/Core/encoder.cpp
Firmware/MotorControl/Core/current.cpp
Firmware/Board/Config/TuneConfig.h
```

## 1. 当前速度估算公式

每 1 ms 更新一次速度：

```text
delta = raw - encoder_prev_raw
speed_sample = delta * CTRL_SPD_EST_HZ * MOT_SENSOR_DIR
speed_fb_diff += (speed_sample - speed_fb_diff) >> CTRL_SPD_FILTER_SHIFT
speed_fb = speed_fb_diff
```

当前：

```text
CTRL_SPD_EST_HZ = 1000
CTRL_SPD_FILTER_SHIFT = 2
```

换成 rpm：

```text
rpm = speed_count_per_s * 60 / MC_SPEED_COUNTS_PER_REV
MC_SPEED_COUNTS_PER_REV = MOT_SENSOR_CPR * MOT_POLE_PAIRS
```

当前：

```text
MC_SPEED_COUNTS_PER_REV = 65536 * 4 = 262144
```

## 2. 差分测速的量化误差

速度来自 1 ms 内 raw 增量。

机械 1 rpm 对应：

```text
262144 / 60 = 4369 count/s
```

在 1 ms 采样窗口里：

```text
1 rpm ~= 4.37 raw count/sample
```

所以低速下 raw 增量很小，量化明显。

当前还有：

```text
CTRL_SPD_POS_DEADBAND = 16
```

如果每个速度样本 raw 增量小于 16 count，就会被当成接近静止。

16 count/sample 约等效：

```text
16 * 1000 * 60 / 262144 ~= 3.7 rpm
```

再加上：

```text
CTRL_SPD_ZERO_SNAP = 500 count/s
```

很小速度会被吸附到 0。

## 3. 低速为什么容易抖

低速控制同时受这些因素影响：

```text
编码器差分量化
速度反馈低通延迟
速度 deadband
zero snap
静摩擦
齿槽转矩
电流采样噪声
死区电压误差
速度 PI 积分
```

所以低速不是简单把 `speed_kp` 加大就好。

典型现象：

```text
speed_fb_rpm 在 0 附近跳
speed_iq_cmd 来回变
speed_pi_integral 慢慢积累
电机一顿一顿
```

## 4. 速度滤波的取舍

当前一阶低通：

```text
new = old + (sample - old) / 4
```

滤波强：

```text
速度更稳
延迟更大
速度环相位裕度变差
```

滤波弱：

```text
响应更快
噪声更大
速度 PI 更容易抖
```

如果低速速度反馈本身量化严重，单纯调滤波不能解决全部问题。

## 5. 速度 spike reject

当前会拒绝异常 raw 差分：

```text
CTRL_SPD_DIFF_SPIKE_RPM = 6500
speed_reject_count
speed_reject_delta
```

位置累计不等到 1 kHz 速度估算才做。当前程序在 20 kHz 角度接受点累计
`encoder_pos`，这样 5000 rpm 时单拍约 1092 count，不会让 1 ms 的 21845
count 直接逼近 `int16_t` 半圈极限。

如果 1 ms 内 raw 跳变超过按 6500 rpm 换算的阈值，会认为是速度采样毛刺：

```text
更新 prev_raw
encoder_pos 不在这里累计
本次 delta 不进入速度低通
speed feedback filter reset
```

这样做是为了把“位置累计”和“速度反馈”分开。位置环依赖 `encoder_pos` 多圈累计，
所以放在更高频的角度链路；速度 PI 不能吃掉明显异常的 1 kHz 速度样本。

如果 `encoder_reject_count` 增长，说明 20 kHz 位置累计发现 raw 单拍跳变超出
当前速度和 hold 时间能解释的范围，本次 raw 不会写进 `encoder_pos`。

如果 `speed_reject_count` 增长，说明速度反馈本身不可信。不要先调速度 PI。

## 6. 速度环输出为什么是 iq

机械方程：

```text
J * d(w)/dt = Te - Tload - B*w
Te ~= Kt * iq
```

所以速度 PI 输出 `iq_ref` 是合理的：

```text
速度误差大 -> 需要更多转矩 -> 增大 iq_ref
```

当前 Speed 模式：

```text
speed PI -> speed_iq_target -> speed_iq_cmd -> iq_ref
```

watch：

```text
speed_err_rpm
speed_iq_target
speed_iq_cmd
iq_ref
iq
```

判断速度环问题时必须看 `iq` 是否真的跟上 `iq_ref`。

## 7. 速度目标斜坡

当前速度目标不是瞬间跳变：

```text
CTRL_SPD_REF_RAMP_RPM_PER_S = 30000
```

速度环 1 kHz 更新，所以约：

```text
30 rpm/sample
```

watch：

```text
speed_ref_rpm
speed_ref_active_rpm
```

如果 `speed_ref_active_rpm` 慢慢追目标，这是正常斜坡，不是命令没生效。

## 8. 速度死区

当前：

```text
CTRL_SPD_CMD_DEADBAND_RPM = 5
```

目标速度在 -5..+5 rpm 内：

```text
reset speed PI
speed_iq_ref = 0
speed_ref_active = 0
speed_deadband_count++
```

目的：

```text
避免零速附近积分保持造成抖动。
```

如果你给很小速度命令，发现 `speed_iq_cmd = 0`，先看是否落入死区。

## 9. 低速调试建议

```text
1. 先在 Current 模式确认小 iq_ref 能平稳输出。
2. 再 Speed 模式给 50 rpm 以上目标，避开极低速量化区。
3. 确认 speed_fb_rpm 符号正确。
4. 限制 iq_limit，避免低速撞击。
5. 看 speed_iq_cmd 是否打满。
6. 看 iq 是否跟随。
7. 最后再调 speed_kp/speed_ki。
```

不要在 `speed_reject_count` 增长或编码器 bad angle 明显时调速度环。

## 10. 低速数据判断表

| 现象 | 更可能原因 | 下一步 |
| --- | --- | --- |
| `speed_fb_rpm` 0 附近跳 | 差分量化/噪声 | 看 encoder_delta 和滤波 |
| `speed_iq_cmd=0` | 目标落入死区 | 看 speed_deadband_count |
| `speed_iq_cmd` 打满 | 速度环要最大转矩 | 看 iq 和 v_limited |
| `speed_reject_count` 增长 | raw 差分异常 | 查编码器/SPI |
| 低速一顿一顿 | 静摩擦、死区、速度 PI | 先降速度 PI 激进度 |
