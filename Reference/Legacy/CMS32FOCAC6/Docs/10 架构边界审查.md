# 10 架构边界审查

本文从架构边界角度审查当前工程，目标是让 `App`、`Board`、`Motor` 三层保持清楚，后续写快环和 FOC 时不把职责搅在一起。

## 当前推荐边界

```text
App
  负责启动、主循环调度、中断入口转发、调试变量输入

Board
  负责本 PCB 的硬件适配：时钟、GPIO、MA600、ADC/PGA、EPWM、输出使能、安全默认状态

Motor
  负责电机控制逻辑：状态机、开环、闭环、FOC、电流/速度/位置环、控制侧保护判断

Vendor
  只放 ARM/Cmsemicon/SEGGER 等外部代码
```

一句话：

```text
App 管程序怎么跑
Board 管板子怎么接
Motor 管电机怎么控
Vendor 管芯片和外部库能力
```

## 当前已经比较干净的地方

### 1. main 已经足够瘦

当前 `main()` 只做：

```c
Board_Init();
Motor_Init();

while (1)
{
    Motor_Set...();
    Motor_TASK();
    Debug_UpdateSnap();
}
```

这是正确方向。`main()` 不再直接关心 ADC/PWM 初始化细节，也不再直接做零漂采样。

### 2. Board_Init 收口正确

当前 `Board_Init()` 内部完成：

```text
时钟
GPIO 安全态
MA600 SPI
ADC/PGA
EPWM
电流零漂
PWM 触发 ADC 同步采样
```

## 2026-05-29：闭环变量边界整理

本次清理目标：

- 删除早期用于验证 PWM 和开环虚拟输出的调试路径。
- 避免 `Board_Analog` 和 `Motor` 中出现同名同义的 `s_iu_cnt/s_iv_cnt/s_iw_cnt`。
- 让电流环快环变量有统一归属，后续接 SVPWM、速度环和位置环时不混乱。

当前边界：

```text
Board_Analog
  保存 ADC raw、offset、offset 后 count 和 adc_sync_count
  对外提供 Board_GetIuCnt/Board_GetIvCnt/Board_GetIwCnt/Board_GetIuvwSum

Motor
  保存控制所需快照和状态
  s_check_cur 用于慢任务安全检查
  s_loop 用于 ADC 中断中的闭环快环计算

App_Debug
  只读 Board 和 Motor，把观察量汇总到 g_debug
```

清理掉的早期调试入口：

```text
debug_open_loop_req
debug_pwm_output_req
Motor_SetOpenLoopReq()
Motor_SetPwmOutputReq()
MOTOR_STATE_OPEN_LOOP
开环虚拟正弦占空比
开环电流峰峰值统计
```

当前只保留一个调试写入口：

```text
debug_motor_enable
```

闭环快环规则：

- `Motor_FastLoop()` 只在 `MOTOR_STATE_CLOSED_LOOP` 且 `s_enable != 0` 时运行。
- 快环内读取本次 ADC 电流快照，保存到 `s_loop.cur`。
- 快环内更新 `angle_raw` 和 `angle_elec`，避免使用主循环中的旧角度。
- 快环内只做 Clarke/Park、PI、InvPark 计算。
- 当前 SVPWM 尚未接入，因此 `s_loop.output_ready = 0`，不打开功率输出。
- 快环内不做阻塞打印、延时、动态内存，也不反复执行硬件关断写操作。

PI 修正：

- `Foc_PiUpdate()` 必须把计算出的 `output` 写回 `pi->output` 再返回。
- PI 积分项使用 `int32_t`，避免后续定点电流环中轻易溢出。

后续接 SVPWM 时的入口：

```text
Motor_FastLoop()
  -> 更新 s_loop.cur
  -> 更新 s_loop.angle_elec
  -> Clarke/Park
  -> PI
  -> InvPark
  -> SVPWM
  -> duty 限幅
  -> s_loop.output_ready = 1
  -> Board_SetPwmDuty()
```

但在接入真实输出前，仍必须先补齐：

- 电流限幅。
- duty 限幅。
- PI 抗饱和。
- MA600 角度跳变保护。
- 丝杠软件限位。
- 明确的输出使能状态和故障复位策略。

## 2026-05-29：架构体检和后续风险

本次重新检查当前 `App/Board/Motor/Docs` 后，当前骨架整体方向是对的：

```text
App
  main、ADC_IRQHandler、Debug_UpdateSnap

Board
  Clock/GPIO/MA600/ADC/PGA/EPWM/安全默认状态

Motor
  状态机、闭环计算骨架、FOC 数学调用、故障关断

Motor_Foc
  纯数学函数和 PI
```

已经顺手收口的点：

- `MotorCurrent_t` 和 `MotorLoop_t` 从公共 `Motor.h` 移回 `Motor.c`，避免外部模块依赖内部控制结构。
- `Board_Analog.c` 中由 ADC 中断更新、主循环读取的电流变量加上 `volatile`。
- [[01 文件地图导航]] 新增为文档地图。

当前会影响后续 FOC 的风险：

1. `Motor_FastLoop()` 里直接调用 `Board_ReadAngleRaw()`，这可能是阻塞 SPI。20 kHz 电流环下必须测一次读角耗时；如果耗时不可接受，要改为角度缓存、分频读取或更快 SPI 时序。
2. `App_Debug` 在主循环中读取 ISR 更新的数据，当前调试可接受，但不是严格原子快照。后续如果要用 FreeMASTER/RTT 做更严肃的波形分析，应设计 `Board_GetCurrentSnap()` / `Motor_GetLoopSnap()` 这类一次性复制接口。
3. 当前 PI 还是最小版本，缺少按采样周期缩放的参数定义、积分抗饱和策略、输出物理量说明。
4. `Motor_FastLoop()` 当前 `id_ref/iq_ref` 都固定为 0，只能验证数学链路，不能代表真实电流闭环。
5. 当前没有 SVPWM、电压矢量限幅、duty 限幅和输出允许状态机，所以保持 `output_ready = 0` 是正确的。
6. 当前 `Motor` 仍直接包含 `Board_Analog.h/Board_PWM.h`，短期适合调试；后续控制稳定后，可以通过 `Board.h` 或结构体快照进一步收口。
7. 丝杠系统还缺少机械零点、软件行程、速度/加速度限制、跟随误差和堵转保护。没有这些之前，不应进入可持续运动的速度环/位置环。

后续建议优先级：

```text
1. 测 MA600 SPI 读角耗时
2. 增加角度 unwrap 和速度估算
3. 为 PI 定义清楚的 Q 格式和单位
4. 实现 SVPWM 只计算不输出
5. 增加 duty/电流/角度跳变保护
6. 设计明确的输出允许和故障复位策略
7. 再做小电流短时间输出

## 仍然有一点模糊但当前可接受的地方

### 1. Motor 直接包含 Board_Analog.h 和 Board_PWM.h

当前 `Motor.c` 直接调用：

```c
Board_GetIuCnt();
Board_GetIvCnt();
Board_GetIwCnt();
Board_GetIuvwSum();
Board_SetPwmDuty();
Board_EnablePwmOutput();
Board_ForcePwmOff();
Board_IsPwmOffSafe();
```

这在当前阶段可接受，因为 `Motor` 正在贴近硬件调试，需要直接观察电流和控制 PWM。

但从更干净的架构看，后续可以把这些接口收成更语义化的 Board 总接口，例如：

```c
Board_GetPhaseCurrent(&current);
Board_SetPhaseDuty(duty_u, duty_v, duty_w);
Board_EnableMotorPower(enable);
Board_IsMotorPowerSafe();
```

这样 `Motor` 不需要知道当前电流来自 ADC CH0/CH2/CH3，也不需要知道 EPWM 的具体模块名。

### 2. App_Debug 直接包含多个 Board 子模块

`App_Debug.c` 当前直接读取 `Board_Analog`、`Board_PWM`、`Motor` 数据，目的是把变量暴露给调试器。

这属于“调试聚合器”，当前可以接受。它天然会跨层读取状态。

注意限制：

- `App_Debug` 只读状态，不改变控制行为。
- 不在 `App_Debug` 中开关 PWM。
- 不在 `App_Debug` 中写外设寄存器。
- 不在中断里调用格式化打印。

### 3. Board_Analog.h 暴露了偏底层的 getter

例如：

```c
Board_GetIuRaw();
Board_GetIuCnt();
Board_GetAdcSyncCount();
```

这些对调试很方便，但不一定适合长期作为 Motor 的主接口。

后续 FOC 稳定后，可以考虑保留这些接口给调试层，给 `Motor` 提供一个更聚合的结构：

```c
typedef struct
{
    int16_t iu_cnt;
    int16_t iv_cnt;
    int16_t iw_cnt;
    int16_t sum_cnt;
} BoardCurrent_t;

void Board_GetCurrent(BoardCurrent_t* current);
```

这会用到 C 里的“结构体指针传参”，比多个 getter 更清楚，也更适合快环。

### 4. Board_PWM.h 暴露了 PWM 寄存器调试读取

`Board_GetPwmDebug()` 当前用于调试很有价值，但它不是控制算法接口。

建议长期规则：

- `Motor` 不调用 `Board_GetPwmDebug()`。
- `App_Debug` 可以调用。
- `Motor` 只调用 `Board_SetPwmDuty()`、`Board_EnablePwmOutput()`、`Board_ForcePwmOff()` 这类控制语义接口。

## 建议立即保持的规则

### 1. Board.h 只放少量总入口

当前 `Board.h` 保持：

```c
void Board_Init(void);
uint8_t Board_HandleAdcIrq(void);
uint16_t Board_ReadAngleRaw(void);
```

这是好的。不要再把 `Board_InitClock()`、`Board_InitGPIO()` 等内部步骤暴露出去。

### 2. App 不直接初始化外设

`main()` 不应该再出现：

```c
Board_CalibCurrentOffset(...);
Board_InitPwmAdcSync();
Board_InitPwm();
ADC_Config...
EPWM_Config...
GPIO_Init...
```

这些都应该留在 `Board`。

### 3. Board 不调用 Motor

不要在 `Board_Analog.c` 或 `Board_PWM.c` 里调用：

```c
Motor_FastLoop();
Motor_TASK();
Motor_Set...
```

当前通过 `ADC_IRQHandler()` 串起来是更干净的方式。

### 4. Motor 不碰 Vendor

`Motor.c` 不应直接包含：

```c
adc.h
epwm.h
gpio.h
ssp.h
CMS32M6510.h
```

如果 Motor 需要硬件数据，走 `Board` 接口。

## 建议下一步收口顺序

### 第一步：快环先做计数和保护

`Motor_FastLoop()` 先只做：

```text
fast_loop_count++
读取当前三相电流
判断采样是否异常
必要时关 PWM / 置故障标志
```

此时不要急着写 Clarke/Park。

### 第二步：引入 BoardCurrent_t

当快环开始频繁读取三相电流时，建议新增：

```c
typedef struct
{
    int16_t iu_cnt;
    int16_t iv_cnt;
    int16_t iw_cnt;
    int16_t sum_cnt;
} BoardCurrent_t;
```

然后用：

```c
void Board_GetCurrent(BoardCurrent_t* current);
```

优点：

- 快环一次拿到一致的一组电流值。
- 函数参数少。
- `Motor` 不需要调用四个 getter。
- 这是学习 C 结构体指针传参的好场景。

### 第三步：把快环状态做成结构体

当前 `Motor` 保持 C 文件。后续快环状态不要再做成零散全局变量，优先用结构体收起来，例如：

```c
typedef struct
{
    uint32_t count;
    int16_t iu;
    int16_t iv;
    int16_t iw;
    int16_t sum;
    uint8_t current_ok;
} MotorFast_t;
```

优点：

- 快环变量集中。
- 调试时更容易看。
- 后续加入 Clarke/Park/PI 时不会散成一堆全局变量。

### 第四步：再做 Clarke/Park

只有当快环计数、电流快照、保护判断都稳定后，再加入：

```text
Clarke
Park
Id/Iq
电流 PI
反 Park
SVPWM
```

## 当前不建议马上做的事

- 不建议马上把 `Board_Analog.h` 全部藏起来，因为当前调试还需要 raw/cnt/offset。
- 不建议马上把 `Motor.c` 拆成很多文件，因为工程还小，过早拆分会增加理解成本。
- 不建议继续引入 C++。已经尝试过 `Motor.cpp`，当前阶段没有比 C 更简单。
- 不建议马上把 MA600 SPI 读角放进 20 kHz 快环，先测耗时。

## 当前架构结论

当前工程已经具备一个干净的骨架：

```text
Board_Init() 让硬件安全可运行
Motor_Init() 初始化控制状态
main() 负责慢任务调度
ADC_IRQHandler() 触发快环
Board 只处理硬件
Motor 只处理控制
App_Debug 只聚合调试观察
```

剩下的主要模糊点不是“目录错了”，而是部分接口还偏调试型。下一步写快环时，不要继续加散乱 getter，优先用结构体把“电流快照”和“快环状态”收起来。

## 2026-05-27：Motor 尝试 C++ 后退回 C

本次先尝试把 `User/Motor/Src/Motor.c` 改为 `User/Motor/Src/Motor.cpp`，随后根据实际可读性退回 C。

尝试 C++ 后发现的问题：

- 当前 `Motor` 逻辑还不复杂，类没有明显减少代码量。
- `MotorController` 和 `extern "C"` 包装增加了额外阅读成本。
- 对硬件调试阶段来说，清楚的 C 状态机比混编边界更直接。

保留的有效收获：

- `App_Debug` 继续通过 `Motor_GetDebug()` 一次取得 `MotorDebug_t` 快照。
- 删除旧的单项 getter：`Motor_GetState()`、`Motor_GetEnable()`、`Motor_GetTargetSpeed()`、`Motor_GetCheck()`、`Motor_GetCurrentStat()`、`Motor_GetOpenLoopDebug()`。
- `Motor.h` 现在只保留控制入口、调试快照和角度换算接口。

当前边界：

- `Board` 仍然是 C，负责硬件。
- `App/main.c` 仍然是 C，负责调度。
- `Motor` 仍然是 C，负责控制状态机。

验证：

- Keil ARMCC5 构建通过。
- 构建结果：`0 Error(s), 0 Warning(s)`。
- 构建日志确认 `Motor.c` 按 C 文件编译。
