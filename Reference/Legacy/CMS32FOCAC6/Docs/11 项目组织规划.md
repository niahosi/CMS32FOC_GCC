## 总体原则

当前主工程目录保持精简，但自写源码统一收入 `User`：

```text
CMS32FOCAC6/
├─ User/
│  ├─ App/
│  ├─ Board/
│  ├─ Config/
│  └─ Motor/
├─ Vendor/
├─ Docs/
│  └─ Hardware/
├─ Objects/
└─ Listings/
```

根目录中 `Reference` 放外部资料和参考工程，`Archive` 放压缩包、编码备份和旧测试。不要在主工程里新增 `Bsp`、`Driver`、`Encoder`、`Control`、`Middleware` 等同级目录。

- `Vendor`：只放厂商和 ARM 提供的文件，尽量不改。
- `User/Config`：放项目可调参数入口，例如电机/编码器参数、控制环参数、保护阈值、PWM 频率和测试默认值。
- `User/Board`：放和这块 PCB 直接相关的外设适配。
- `User/Motor`：放电机状态机、FOC、开环/闭环控制。
- `User/App`：放程序入口、任务调度和应用策略。

一句话：`Vendor` 管芯片能力，`Config` 管可调参数，`Board` 管这块板子怎么接线，`Motor` 管电机怎么控制，`App` 管程序怎么运行。

## Board 层放什么

MA600、Hall、NTC、电流采样、EPWM 输出、刹车保护、调试口选择都属于 `Board`。

原因是它们都和本板原理图强绑定：

- MA600 固定接在 `P02/P03/P04/P05`。
- Hall 固定接在 `P21/P22/P23`。
- NTC 固定接在 `P20/AN4`。
- 三相电流采样固定接在 `P00/P01`、`P24/P25`、`P26/P27`。
- EPWM 直接驱动芯片内部 3P3N 功率输出。
- `P06/P07` 同时涉及 SWD 和 UART 调试取舍。

所以这些不应该放进 `Motor`，也不应该直接散落在 `App/main.c`。

## 推荐文件规划

在不改变文件夹结构的前提下，后续可以逐步形成这些文件：

```text
User/
├─ Config/
│  └─ Config.h
├─ Board/
│  ├─ Inc/
│  │  ├─ Board.h
│  │  ├─ Board_Config.h
│  │  ├─ Board_MA600.h
│  │  ├─ Board_Analog.h
│  │  └─ Board_PWM.h
│  └─ Src/
│     ├─ Board.c
│     ├─ Board_MA600.c
│     ├─ Board_Analog.c
│     └─ Board_PWM.c
├─ Motor/
│  ├─ Inc/
│  │  ├─ Motor.h
│  │  ├─ Motor_Foc.h
│  │  ├─ Motor_Loop.h
│  │  └─ Motor_Diag.h
│  └─ Src/
│     ├─ Motor.c
│     ├─ Motor_Foc.c
│     ├─ Motor_Loop.c
│     └─ Motor_Diag.c
└─ App/
   ├─ Inc/
   │  └─ App_Debug.h
   └─ Src/
      ├─ main.c
      └─ App_Debug.c
```

这是当前工程已经采用的形态。物理路径带 `User`，但 Keil 工程里的 group 名继续保留 `App / Board / Motor`，这样工程树仍然短。注意文件名当前使用 `Board_PWM.h/.c`，不要在文档或工程里再混用 `Board_Pwm.h/.c`。

## Board 文件职责

### Board.h

`Board.h` 是对外总入口。`App` 和 `Motor` 优先包含这个文件，而不是直接包含很多具体外设头文件。

当前 `Board.h` 已经有这些接口：

```c
void Board_Init(void);
uint8_t Board_HandleAdcIrq(void);
uint16_t Board_ReadAngleRaw(void);
```

内部初始化顺序由 `Board.c` 的 `static` 函数收口，不再从 `Board.h` 对外暴露。

### Config.h 和 Board_Config.h

`User/Config/Config.h` 是当前工程的可调参数入口，适合放“调试时经常会改、但不属于寄存器细节”的参数。

适合放：

```c
#define MOTOR_POLE_PAIRS
#define MOTOR_ELEC_ZERO
#define MOTOR_CURRENT_KP
#define MOTOR_CURRENT_KI
#define MOTOR_CURRENT_REF_LIMIT
#define PWM_FREQ_HZ
#define PWM_DUTY_MIN
#define PWM_DUTY_MAX
#define ADC_TO_AMP
```

不适合放：

```text
ADC 通道号、EPWM 通道号、GPIO 复用、MA600 寄存器命令、芯片寄存器位定义
```

这些属于 Board 层或 Vendor 层实现细节，继续留在对应 `.c/.h` 内。这样 `Config.h` 不会变成大杂烩，后续调电流环、速度环、保护阈值时也能快速找到入口。

`Board_Config.h` 当前只保留为兼容转发头文件，内部包含 `Config.h`。后续新代码优先直接包含 `Config.h`。

### Board_MA600.h / Board_MA600.c

MA600 不单独做 `Encoder` 层，而是作为板载外设放在 `Board` 里。

职责：

- 初始化 `P02/CNS` 片选。
- 初始化 `P03/P04/P05` SPI。
- 读取 MA600 16 bit 原始数据。
- 提供原始角度、方向、零点偏移。
- 只处理和 MA600 通信、安装校准有关的事。

不在这里做完整 FOC，也不在这里决定目标速度。

当前建议接口：

```c
void Board_InitMA600Spi(void);
uint16_t Board_MA600_ReadRaw16(void);
uint16_t Board_MA600_ReadAngleRaw(void);
void Board_MA600_SetZeroOffset(uint16_t offset);
void Board_MA600_SetDirection(int8_t direction);
```

对外如果想更简单，可以由 `Board.h` 再包一层：

```c
uint16_t Board_ReadMA600RawAngle(void);
```

### Board_Analog.h / Board_Analog.c

模拟量相关都放这里。

职责：

- ADCLDO/ADC 初始化。
- PGA0/1/2 初始化。
- 三相电流 ADC 原始值读取。
- 零电流偏置采集。
- NTC ADC 读取。
- 必要时处理 ADC 触发源和采样时序。

当前对外入口先从 `Board.h` 走：

```c
void Board_InitAdcCur(void);
void Board_Analog_CalibrateCurrentOffset(void);
void Board_Analog_ReadPhaseCurrentRaw(int16_t *iu, int16_t *iv, int16_t *iw);
uint16_t Board_Analog_ReadNtcRaw(void);
```

初期没有板子时，可以先返回固定假数据。后续如果 `Board_Analog.c` 里函数变多，可以再把内部接口整理成 `Board_Analog_*` 风格。

### Board_PWM.h / Board_PWM.c

PWM 和功率输出保护放这里。

职责：

- EPWM 初始化。
- 三相占空比更新。
- 输出使能/关闭。
- Brake/Mask/Inactive 状态。
- 死区、极性、中心对齐模式。

当前 `Board.h` 已经保留了这类接口：

```c
void Board_InitPwm(void);
void Board_SetMotorOutputEnable(uint8_t enable);
void Board_SetPhaseDuty(uint16_t duty_u, uint16_t duty_v, uint16_t duty_w);
```

重要规则：上电默认关闭输出。除非电流采样、比较器保护、PWM 极性和死区都确认，不允许在初始化里直接打开功率输出。

### 调试口

当前没有单独保留 `Board_Debug.c`。调试口策略先记录在文档和 `Board.c` 初始化约束中。

职责：

- 管理 `P06/P07` 是保留 SWD，还是后续切成 UART。
- 后续如果真的要切 UART，再新增明确的 Board 调试口文件。

开发阶段建议保留 SWD。等程序稳定、确认有恢复下载方法后，再考虑把 `P06/P07` 切成串口。

## Motor 层放什么

`Motor` 不直接碰寄存器，不直接配置 GPIO，不直接调用厂商 `ssp.c/gpio.c/epwm.c/adc.c`。

`Motor` 只通过 `Board` 拿传感器数据和输出 PWM：

```c
angle = Board_ReadMA600RawAngle();
Board_ReadPhaseCurrentRaw(&iu, &iv, &iw);
Board_SetPhaseDuty(duty_u, duty_v, duty_w);
```

`Motor.c` 当前负责：

- 电机状态机。
- 角度方向和零点处理。
- Clarke/Park。
- 电流环计算骨架。
- 故障状态处理。

当前状态机：

```text
MOTOR_STATE_IDLE
MOTOR_STATE_SENSOR_CHECK
MOTOR_STATE_CLOSED_LOOP
MOTOR_STATE_FAULT
```

早期开环虚拟正弦输出路径已经删除。后续如需对齐，应作为 `ALIGN` 子模式重新设计，而不是恢复旧的 `OPEN_LOOP` 调试路径。

## App 层放什么

`App/main.c` 保持简单。

当前结构：

```c
int main(void)
{
    Board_Init();
    Motor_Init();

    while (1)
    {
        Motor_SetEnable(debug_motor_enable);
        Motor_TASK();
        Debug_UpdateSnap();
    }
}
```

`App` 不负责某个引脚怎么复用，也不直接读 MA600 SPI。

## Vendor 层放什么

`Vendor` 只放外部来源代码：

- ARM CMSIS Core。
- Cmsemicon Device。
- Cmsemicon Driver。

原则：

- 不在 `Vendor` 里写自己的板级逻辑。
- 不随手改厂商驱动。
- 如果必须修厂商 bug，要在文档里记录原因。

## MA600 数据流

MA600 的数据建议这样流动：

```text
Vendor/ssp.c
    ↓
Board_MA600.c
    ↓
Board_ReadMA600RawAngle()
    ↓
Motor.c
    ↓
FOC 电角度
```

不要让 `Motor.c` 直接调用 `SSP_MasterSendReceive()` 之类的厂商函数。这样以后 MA600 SPI 模式、片选时序、校准方式变了，只改 `Board_MA600.c`。

## 未到板前可以先写什么

没有控制板时，优先写这些：

1. `Board.h` 对外接口。
2. `Board.c` 安全初始化顺序。
3. `Board_MA600.c` 的假数据读取。
4. `Board_Analog.c` 的假电流、假 NTC。
5. `Board_PWM.c` 的空输出函数，默认关闭输出。
6. `Motor.c` 状态机。
7. `Motor.c` 中角度、速度、电流变量结构。

暂时不要写真实励磁逻辑。

## 第一阶段最小落地顺序

第一阶段只做能编译、能组织代码：

```text
Board.h
Board.c
Board_Config.h
Motor.h
Motor.c
main.c
```

第二阶段再拆：

```text
Board_MA600.h/.c
Board_Analog.h/.c
Board_PWM.h/.c
```

第三阶段等板子到手后再接真实外设：

```text
MA600 SPI 实测
ADC 零漂实测
NTC 实测
Hall 实测
EPWM 波形实测
低占空比开环
FOC 闭环
```

## C 和 C++ 使用约定

当前工程保持 C 为主：

- `User/Board` 使用 C。
- `Vendor` 保持 C。
- `User/App/main.c` 使用 C。
- `User/Motor` 使用 C，核心入口文件为 `User/Motor/Src/Motor.c`。

本项目已经试过把 `Motor.c` 切成 `Motor.cpp`，但当前阶段没有带来明显简化：

- `MotorController` 类和 C 包装函数增加了额外层级。
- Keil/ARMCC5 混编需要维护 `extern "C"` 边界，当前收益不够。
- 对现在的状态机来说，C 的 `static` 函数、`static` 状态和结构体快照已经够清晰。

当前保留的简化点：

- `Motor_GetDebug()` 一次返回 `MotorDebug_t`，避免 `App_Debug` 调用一串单项 getter。
- 删除不再需要的 `Motor_GetState()`、`Motor_GetCheck()`、`Motor_GetOpenLoopDebug()` 等旧调试 getter。
- 后续快环状态优先用 `typedef struct` 聚合，而不是继续增加散乱全局变量。

以后如果再次考虑 C++，判断标准不是“能不能用”，而是它是否让代码更短、更少接口、更难误用。当前结论是：先不使用 C++。
