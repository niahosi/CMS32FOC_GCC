# 09 文件拆分与分层规划

本文回答两个问题：

1. 当前 `Motor.c` / `Motor.h` 是否已经太大，是否需要马上拆分。
2. 未来 `App`、`Motor`、`Board` 三层的文件应如何增长，避免越写越乱。

## 1. 当前结论

截至 2026-05-29：

```text
User/Motor/Src/Motor.c    约 308 行
Motor/Inc/Motor.h    约 104 行
Board_Analog.c       约 235 行
Board_PWM.c          约 187 行
App/main.c           约 57 行
```

当前 `Motor.c` 还没有大到必须拆。它现在主要包含：

- 电机状态机。
- 传感器/电流安全检查。
- 闭环快环计算骨架。
- 调试快照输出。
- 电角度方向修正。

这些内容目前还可以放在一个文件中，优点是上下文集中，适合当前学习和快速修改。

当前不建议立刻拆源码，原因：

- FOC 还只是计算骨架，SVPWM、速度环、位置环还没有真正写入。
- 过早拆分容易产生很多很薄的文件，反而增加跳转成本。
- 现在更重要的是把职责边界和拆分标准写清楚，等代码真的长起来再按标准拆。

## 2. 什么时候必须拆 Motor

满足任意两条时，就应该拆：

- `Motor.c` 超过约 `500` 行。
- `Motor.c` 同时包含状态机、FOC 数学、速度环、位置环、丝杠轨迹。
- `Motor_FastLoop()` 周围出现大量只服务于 FOC 的静态函数。
- `Motor_TASK()` 里的状态机和算法细节互相穿插，阅读时需要反复上下跳。
- `Motor.h` 暴露了太多内部结构，而不是少量公共接口和调试快照。
- 新增一个功能时，不确定应该放在 `Motor.c` 哪个区域。

拆分目标不是“文件越多越专业”，而是：

```text
状态机看状态机
快环看快环
数学看数学
轨迹看轨迹
硬件看 Board
调度看 App
```

## 3. Motor 推荐增长路线

### 当前阶段

保持：

```text
Motor/
├─ Inc/
│  ├─ Motor.h
│  └─ Motor_Foc.h
└─ Src/
   ├─ Motor.c
   └─ Motor_Foc.c
```

职责：

- `Motor.h`：对 App 暴露的 Motor 公共接口、状态枚举、调试快照。
- `Motor.c`：状态机、闭环计算骨架、Motor 内部状态。
- `Motor_Foc.h/.c`：Clarke/Park/PI/InvPark/SVPWM 等纯算法函数。

### 写入 SVPWM 后

如果 `Motor_Foc.c` 继续增长，优先仍放在 `Motor_Foc.c` 中：

```text
Motor_Foc.c
  Foc_Sin/Cos
  Foc_ClarkeTransform
  Foc_ParkTransform
  Foc_InvParkTransform
  Foc_PiUpdate
  Foc_Svpwm
  Foc_VoltageLimit
```

不要把 SVPWM 写进 `Board_PWM.c`。`Board_PWM.c` 只知道“写三个 duty count”，不应该知道电压矢量、扇区、零序注入。

### 写入速度环后

当速度估算和速度 PI 稳定后，可新增：

```text
Motor/
├─ Inc/
│  └─ Motor_Speed.h
└─ Src/
   └─ Motor_Speed.c
```

职责：

- 角度 unwrap。
- 速度差分。
- 速度滤波。
- 速度 PI。
- 速度目标限幅。

如果速度功能只有几十行，先留在 `Motor.c`；超过约 `150` 行或需要独立调参时再拆。

### 写入丝杠位置环后

位置环与丝杠安全关系很强，建议单独文件：

```text
Motor/
├─ Inc/
│  └─ Motor_Profile.h
└─ Src/
   └─ Motor_Profile.c
```

职责：

- 机械零点。
- 机械圈数到 mm 的换算。
- 软件行程限位。
- 梯形速度规划。
- 最大速度/最大加速度限制。
- 跟随误差保护。
- 堵转或卡滞判断。

位置环不要写进 `Board`，因为它不是硬件驱动；也不要写进 `App`，因为它属于电机控制策略。

### 参数逐渐增多后

当宏定义明显变多，建议新增：

```text
Motor/
├─ Inc/
│  └─ Motor_Param.h
```

职责：

- 电机极对数。
- 磁环极对数。
- 电角度零点。
- 电流限幅。
- 速度限幅。
- 位置限幅。
- PI 默认参数。
- 弱磁默认参数。

当前 `Motor.h` 中的参数宏还不多，可以暂时保留。等宏超过约 `25` 个，或者开始出现速度/位置/弱磁多类参数，再拆到 `Motor_Param.h`。

## 4. App 文件规划

当前保持：

```text
App/
├─ Inc/
│  ├─ main.h
│  └─ App_Debug.h
└─ Src/
   ├─ main.c
   └─ App_Debug.c
```

职责：

- `main.c`：初始化、主循环、ADC 中断入口。
- `App_Debug.c`：把 Board 和 Motor 的状态聚合成 `g_debug`。

后续如果中断入口变多，可以新增：

```text
App/Src/App_Irq.c
App/Inc/App_Irq.h
```

拆分条件：

- 出现多个中断入口，例如 ADC、UART、Timer、Fault。
- `main.c` 超过约 `150` 行。
- `main.c` 里除了调度还开始出现大量中断细节。

后续如果有上位机命令，可以新增：

```text
App/Src/App_Command.c
App/Inc/App_Command.h
```

职责：

- 解析调试命令。
- 设置 Motor 目标值。
- 故障复位命令。
- 参数保存/加载命令。

但当前只有 `debug_motor_enable`，不需要拆。

## 5. Board 文件规划

当前保持：

```text
Board/
├─ Inc/
│  ├─ Board.h
│  ├─ Board_Config.h
│  ├─ Board_MA600.h
│  ├─ Board_Analog.h
│  └─ Board_PWM.h
└─ Src/
   ├─ Board.c
   ├─ Board_MA600.c
   ├─ Board_Analog.c
   └─ Board_PWM.c
```

职责：

- `Board.h`：板级总入口，只暴露少量通用接口。
- `Board.c`：初始化顺序、顶层板级转发。
- `Board_Config.h`：板级固定参数。
- `Board_MA600.c`：MA600 SPI 和寄存器读写。
- `Board_Analog.c`：ADC/PGA/零漂/同步采样。
- `Board_PWM.c`：EPWM、输出使能、关断、死区、duty 写入。

后续可新增：

```text
Board/Inc/Board_Protect.h
Board/Src/Board_Protect.c
```

拆分条件：

- ACMP/硬件过流刹车开始接入。
- 欠压/过压/过温统一判断开始变多。
- `Board_PWM.c` 中保护逻辑明显超过 PWM 输出控制本身。

后续可新增：

```text
Board/Inc/Board_Temp.h
Board/Src/Board_Temp.c
```

拆分条件：

- NTC 换算、滤波、温度阈值、降额策略开始超过约 `100` 行。

后续可新增：

```text
Board/Inc/Board_DebugPort.h
Board/Src/Board_DebugPort.c
```

拆分条件：

- 真的要把 `P06/P07` 从 SWD 复用为 UART。
- 需要统一管理 RTT/UART/FreeMASTER 通信口。

开发阶段不建议为了“完整”提前创建这些文件。

## 6. 头文件设计规则

### Motor.h

只放：

- `Motor_Init()`
- `Motor_TASK()`
- `Motor_FastLoop()`
- `Motor_SetEnable()`
- `Motor_GetDebug()`
- 状态枚举。
- 对外调试结构体。

不要放：

- `MotorLoop_t` 这类内部运行结构。
- 只给 `Motor.c` 内部使用的私有函数。
- 大量参数宏。
- Board 细节。

### Motor_Foc.h

只放纯算法接口：

- Clarke/Park。
- PI。
- SVPWM。
- 限幅。
- sin/cos。

不要放：

- 电机状态机。
- Board 接口。
- 调试变量。

### Board.h

只放板级总入口：

- `Board_Init()`
- `Board_HandleAdcIrq()`
- `Board_ReadAngleRaw()`

随着后续收口，可考虑增加：

```c
void Board_GetCurrentSnap(BoardCurrent_t* cur);
void Board_SetMotorDuty(uint16_t u, uint16_t v, uint16_t w);
void Board_SetMotorOutput(uint8_t enable);
```

但不要一次性把所有 `Board_Analog` / `Board_PWM` 内部接口都塞进去。

## 7. 拆分时的 Keil 注意事项

新增 `.c` 文件后，必须在 Keil 工程里添加：

```text
Project 窗口右键分组
  -> Add Existing Files to Group
  -> 选择新 .c
```

否则 VS Code 能看到文件，但 Keil 不会编译。

新增 `.h` 文件后，确认：

- 文件位于已有 include path 下。
- 或者在 Keil `Options for Target -> C/C++ -> Include Paths` 中加入对应路径。

当前已有：

```text
Motor/Inc
Board/Inc
App/Inc
```

## 9. 当前行动建议

当前不要马上拆 `Motor.c`。

下一步更合理的是：

1. 继续在 `Motor_Foc.c` 补 SVPWM 和电压限幅。
2. `Motor.c` 只调用 FOC 结果，不把 SVPWM 数学展开在状态机里。
3. 当速度估算超过约 `100~150` 行时，新增 `Motor_Speed.c`。
4. 当丝杠位置规划开始写时，新增 `Motor_Profile.c`。
5. 每新增一个重要文件，同步更新：
   - [[01 文件地图导航]]
   - [[11 项目组织规划]]
   - 必要时更新 [[10 架构边界审查]]

## 10. 函数指针表和回调的使用边界

当前 `Motor_TASK()` 使用 `switch (s_state)` 是合适的，因为状态数量少，调试时可以一眼看清状态流转。

后续可以保留 `MotorState_t` 状态枚举，同时引入函数指针表，把每个状态的入口、运行、退出动作收口到表中。这样不是为了炫技，而是为了让状态机在状态变多以后仍然清晰。

推荐使用位置：

- `Motor` 状态机：适合使用“状态枚举 + 状态处理函数表”。
- `Motor_Foc` 算法策略：以后如果需要在开环、闭环、弱磁、不同 SVPWM 策略之间切换，可以用函数指针选择算法。
- `App_Command` 命令分发：以后有上位机命令时，适合用命令表代替很长的 `if/switch`。
- `Board` 设备抽象：只有在同类设备超过一个时才使用，例如未来同时支持 MA600 和另一种编码器。

当前不建议使用位置：

- 不要把 `Board_SetPwmDuty()`、`Board_ForcePwmOff()` 这类唯一硬件动作包成回调。
- 不要在 ADC 中断快环里层层回调，除非能确定调用链短、耗时稳定。
- 不要为了“面向对象”把所有函数都放进函数指针结构体。

推荐的未来状态表形态：

```c
typedef void (*MotorStateFn_t)(void);

typedef struct
{
    MotorState_t state;
    MotorStateFn_t enter;
    MotorStateFn_t run;
    MotorStateFn_t exit;
} MotorStateNode_t;
```

这种写法的好处：

- 状态数量增加后，新增状态只需要补一个状态节点。
- 每个状态的逻辑天然分开，`Motor_TASK()` 会更短。
- 状态入口动作和周期动作不容易混在一起，例如进入故障时只清一次 PI，而不是每次循环都清。
- 可以更容易做状态跳转记录和调试统计。

这种写法的代价：

- 初期代码会比 `switch` 多几个函数和一张表。
- 调试时调用路径会多一层跳转。
- 函数指针无法像普通直接调用那样一眼看出最终执行函数，新手阶段需要配合命名和文档。

因此本项目建议：

```text
现在：继续使用 switch 状态机
状态超过 5 个或 Motor_TASK 明显变长后：切换为状态函数表
算法策略出现多个版本后：局部使用函数指针
Board 唯一硬件动作：继续直接函数调用
```
